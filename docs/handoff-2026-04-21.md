### 1.4 — Secondary bugs discovered, flagged, not fixed

**Advisory lock doesn't pin a postgres-js connection** (P2 in `TODOS.md`). Shipped with Sedgewick. `db.execute(sql\`SELECT pg_try_advisory_lock(...)\`)`and the corresponding unlock can land on different pool connections because postgres-js rotates connections per query. Net effect: release returns false on a connection that doesn't hold the lock; the real lock persists on the acquiring connection until`idle_timeout: 20s`expires the session. Under cron-only load (1 fire/week) it's probably benign — the 20s idle timeout will release naturally. Under concurrent curls or future load, it would surface as spurious "another weekly-crawl in progress" skips. The code comment at`weekly-crawl/route.ts:128`is also wrong — Postgres releases session locks on session END, not on pool return. Fix pattern:`sql.reserve()`(postgres-js 3.4+) or`sql.begin()` for transaction-scoped pinning. Needs a real-pool test.
