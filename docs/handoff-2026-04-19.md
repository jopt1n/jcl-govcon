## Decisions made

### Decision: `audit_log.contract_id` is nullable + `onDelete: 'set null'`, not cascade

- **Chose:** nullable FK with SET NULL cascade
- **Rationale:** the whole point of an audit log is to answer "what did I do?" even when the source contract is deleted. Orphaned rows keep their `action` + `created_at` as forensic records.
- **Rejected:** cascade delete (loses history), no FK (loses integrity for live contracts)
- **Reversible?** Yes — ALTER can widen/narrow the FK definition later.
- **Could go wrong:** if we ever add a viewer that queries audit_log grouped by contract, orphaned NULLs will pool in a bucket. Add a `WHERE contract_id IS NOT NULL` filter to the viewer query when we build one.

### Decision: PATCH UPDATE + audit_log INSERT wrapped in `db.transaction(async (tx) => ...)`

- **Chose:** atomic transaction; both ops use `tx`, never the outer `db`
- **Rationale:** audit trail must be reliable — a contract that flipped to `promoted=true` without an audit row is worse than either op failing visibly.
- **Rejected:** two separate queries (inconsistent state on partial failure), CAS-style WHERE-clause (premature for v1 single-user)
- **Reversible?** Yes — can be unwrapped to two separate calls anytime.
- **Could go wrong:** if someone refactors and accidentally writes `await db.insert(auditLog)...` inside the transaction callback, the INSERT runs on a separate connection and atomicity silently breaks. There's a unit test (`mockTxUpdateCalled`) asserting the transaction path is exercised.

### Decision: `useNavCounts` will use `Promise.allSettled`, not `Promise.all`

- **Chose:** allSettled, so one fetch rejection doesn't null out the other badge
- **Rationale:** sidebar polls two independent endpoints; if one fails, the other should still update.
- **Rejected:** `Promise.all` (all-or-nothing; one failure blanks both), two independent useEffect hooks (doubles poll boilerplate)
- **Reversible?** Yes, one-line change.
- **Could go wrong:** if the allSettled result handler doesn't check `status === "fulfilled"` on each promise before reading `.value`, a rejected promise throws at read time. Next session is building this — check the test for the rejected path.

### Decision: State-exclusive rendering for Kanban card's gold border

- **Chose:** when `promoted=true`, ONLY `border-l-[4px] border-l-[var(--chosen)]` are in the DOM. The default 3px + classification-color classes are OMITTED.
- **Rationale:** Tailwind specificity depends on CSS order in the emitted stylesheet; layering "default + override" classes is fragile.
- **Rejected:** additive override classes
- **Reversible?** Yes, trivial refactor.
- **Could go wrong:** Commit 3's test asserts exclusivity via `.not.toContain("border-l-[3px]")`. If future design uses `border-l-[3.5px]`, the test passes but the intent is violated. Acceptable for now.

### Decision: Tailwind alpha-modifier on CSS vars is unusable in this codebase

- **Chose:** use precomputed rgba tokens (`--accent-20`, `--chosen-border`) in place of `var(--accent)/20`
- **Rationale:** Tailwind 3.4's alpha modifier silently emits invalid CSS for CSS var references; 10 sites were already broken before this session, visually masked by surrounding design. Verified via `getComputedStyle` that they produced `rgba(0,0,0,0)` transparent.
- **Rejected:** inline `color-mix()` arbitrary values (works but verbose at 10+ use sites), runtime CSS-in-JS (overkill)
- **Reversible?** Yes, but don't; the tokens are the correct long-term shape.
- **Could go wrong:** a future developer reaches for `bg-[var(--X)]/30` syntax because it's the Tailwind idiom. The `globals.css` comment explains why it's broken and directs them to the tokens. Keep that comment.

### Decision: `removeFromInbox` is an inner function in `InboxPage`, not a module-level helper

- **Chose:** closure-based, closes over `setMarking`/`setGroups`/`fetchGroup`
- **Rationale:** at 3 pieces of state, passing them as arguments (4-arg signature) is uglier than closing over them. Option 1 from the Commit 3 eng review's C1 decision.
- **Rejected:** standalone helper taking all state as params (noisy call sites), custom hook (overkill for one consumer)
- **Reversible?** Yes.
- **Could go wrong:** if a third caller shows up in a different component, this won't scale. The pattern on `/chosen` should duplicate rather than try to share — different state shape, different file, <5 lines of overlap.
