# Plan: CHOSEN tier — user-driven promotion above AI's GOOD

Dynamic working name in code: **promoted** (boolean flag). Displayed to the user as **"Great"** (verb: "Promote to Great") and on its own page as **/chosen**. Color token: **gold**.

## Architectural decision: Option B (promoted boolean)

Adopting your recommendation. Justification:

1. **Data integrity.** The AI classification is the durable signal from the model — it's what powers the "was the classifier right?" analytics loop the memory file explicitly calls out (recall-over-precision). Overwriting GOOD → CHOSEN destroys that signal. A boolean `promoted` keeps the AI's answer intact and records the human decision orthogonally.
2. **Blast radius.** Option A touches ~38 hardcoded classification sites, a Postgres `ALTER TYPE` (which requires a DB round-trip outside the Drizzle `push` flow), and the AI prompt + classifier variants. Option B touches roughly **7** files with additive changes.
3. **Precedent.** The schema already has two orthogonal boolean/timestamp axes: `userOverride` (did the human change the AI's mind?) and `reviewedAt` (did the human triage this?). `promoted` / `promotedAt` sits naturally next to them: did the human elevate this one?
4. **Reversibility.** A "demote" is one boolean flip; contracts flow back to their original classification column with no data loss. An enum approach has to remember the "previous" value explicitly.
5. **No AI-prompt surgery.** Grok keeps emitting GOOD/MAYBE/DISCARD. There is zero risk of the AI outputting CHOSEN, zero risk of the recall-over-precision policy drifting.
6. **Composition with status.** The pipeline status enum (IDENTIFIED → WON) is already orthogonal to classification. `promoted` adds a third orthogonal dimension in exactly the same style.

Only case where Option A wins is if we want a single sortable column in SQL (`ORDER BY classification`). We don't — we already sort by `promoted DESC, posted_date DESC` trivially.
